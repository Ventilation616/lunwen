# AI智能运动动作矫正系统架构设计说明书

**文档版本**: v1.0
**日期**: 2026-03-01

## 1. 项目背景与目标

随着全民健身意识的觉醒与居家运动需求的激增，传统健身模式中缺乏专业指导、动作不规范导致运动损伤等问题日益凸显。现有市场上的运动辅助工具多侧重于简单的计数或视频跟练，缺乏对用户动作质量的实时反馈与矫正能力。

本项目旨在设计并实现一套基于计算机视觉与人工智能技术的“AI智能运动动作矫正系统”。该系统通过集成人体姿态估计（Pose Estimation）与大语言模型（LLM）技术，构建具备“动作捕捉-实时分析-智能反馈”闭环能力的数字化健身平台。核心目标在于利用机器视觉算法精准识别用户运动姿态，提供毫秒级的动作纠错指导；同时结合生成式AI打造个性化虚拟私教，实现运动数据的科学量化与健身知识的普及，从而降低居家运动门槛，保障运动安全，提升健身效果。

## 2. 系统整体架构说明

本系统采用 **C/S (Client/Server)** 架构与 **B/S (Browser/Server)** 混合模式，前端基于微信小程序生态，后端采用轻量级 Web 服务框架，结合云端数据库与第三方 AI 服务，形成“端-云-边”协同的整体架构。

系统主要分为四层：
1.  **感知与交互层 (Client Tier)**：由微信小程序构成，负责视频流采集、用户交互界面展示、实时反馈渲染及基础数据缓存。
2.  **管理与运营层 (Admin Tier)**：新增 **Web 后台管理系统**，面向管理员与运营人员，提供用户管理、动作库维护、模型监控及数据分析可视化功能。
3.  **业务与计算层 (Application Tier)**：部署于云服务器，承载核心计算任务，包括人体骨骼关键点检测、动作逻辑判定、AI 对话中转处理及科普内容分发。
4.  **数据与服务层 (Data & Service Tier)**：包含微信云数据库（存储用户档案与运动日志）、对象存储（临时图像缓冲）及外部大模型 API 服务（DeepSeek）。

## 3. 技术选型及理由

| 模块 | 技术选型 | 选型理由 |
| :--- | :--- | :--- |
| **前端交互 (C端)** | **微信小程序 (Native)** | 依托微信庞大生态，无需下载安装，即用即走；提供丰富的原生组件与相机 API，便于快速实现视频流采集与 UI 交互。 |
| **后台管理 (B端)** | **Vue 3 + Element Plus** | 主流的现代化前端技术栈，组件丰富，开发效率高，适合构建复杂的数据可视化大屏与表单管理界面。 |
| **后端框架** | **Python + Flask** | Python 在 AI 与数据处理领域生态成熟；Flask 框架轻量灵活，易于集成 OpenCV 与 MediaPipe 等视觉库，适合快速构建 RESTful API。 |
| **视觉算法** | **MediaPipe + OpenCV** | MediaPipe 提供高性能的端侧/移动端人体姿态估计方案，延迟低、精度高，适合实时性要求高的动作矫正场景；OpenCV 用于基础图像预处理。 |
| **AI 大模型** | **DeepSeek API** | 具备强大的自然语言理解与生成能力，能够根据用户输入提供专业的健身建议，且 API 接入成本相对可控。 |
| **数据存储** | **微信云开发 (CloudBase)** | 免运维的 Serverless 数据库方案，与小程序天然集成，鉴权方便，支持高并发读写，适合存储非结构化的运动日志与用户配置。 |

## 4. 分层设计 (Layered Architecture)

为了保证系统的可维护性与扩展性，后端服务遵循关注点分离原则，采用经典的分层架构设计：

### 4.1 表现层 (Controller Layer)
*   **职责**：负责接收前端 HTTP 请求，解析参数，进行基础校验，并调用下层服务处理业务，最后封装统一格式的 JSON 响应返回给前端。
*   **核心组件**：
    *   `PoseController`: 处理姿态检测请求 (`/detect`, `/situp`)。
    *   `UploadController`: 处理图片上传请求 (`/upload`)。
    *   `AIChatController`: 处理与 DeepSeek 的对话请求 (`/seedToDeepSeek`)。

### 4.2 业务逻辑层 (Service Layer)
*   **职责**：核心业务逻辑的实现场所，负责编排算法调用、数据处理流程及第三方服务交互。
*   **核心组件**：
    *   `PoseAnalysisService`: 封装 `MediaPipe` 调用逻辑，实现 `findPose`, `findPosition`, `findAngle` 等核心算法，判断动作是否标准（如俯卧撑角度计算、仰卧起坐状态判定）。
    *   `LLMService`: 封装 DeepSeek API 的调用逻辑，管理 Prompt 模板（如设置“健身教练”人设），处理上下文对话。
    *   `FileStorageService`: 管理临时文件的保存、读取与清理逻辑。

### 4.3 领域层 (Domain Layer)
*   **职责**：定义核心业务实体与规则，不依赖于具体的技术实现。
*   **核心实体**：
    *   `User`: 用户基本信息实体。
    *   `SportRecord`: 运动记录实体（包含类型、个数、错误详情、时间戳）。
    *   `PoseData`: 姿态数据实体（关键点坐标集合、关节角度）。
    *   `AnalysisResult`: 动作分析结果实体（是否标准、错误类型）。

### 4.4 数据访问层 (Repository Layer)
*   **职责**：负责与数据库或外部存储系统进行交互，屏蔽底层数据存储细节。
*   **核心组件**：
    *   `SportRecordRepository`: 负责运动记录的 CRUD 操作（连接微信云数据库）。
    *   `KnowledgeRepository`: 负责科普文章的查询与检索。

### 4.5 后台管理子系统设计 (Admin Subsystem)

为了支撑系统的长期运营与数据分析，设计精简高效的后台管理系统，包含以下四大核心模块：

1.  **主页 (Dashboard)**
    *   **功能**：
        *   **实时数据大屏**：展示当前在线人数、今日运动总次数。
        *   **用户活跃度漏斗分析**：展示用户从访问 -> 注册 -> 运动 -> 达标的转化率。
        *   **动作矫正成功率趋势图**：按时间轴展示用户动作达标率的变化趋势。
        *   **模型准确率追踪报表**：展示 AI 模型在不同场景下的置信度与准确率统计。
    *   **价值**：提供全局视角的运营监控，快速发现系统异常与业务增长点。

2.  **运动动作库管理 (Motion Library)**
    *   **功能**：
        *   **动作管理**：动作类型的增删改查（如俯卧撑、深蹲）。
        *   **标准定义**：上传标准动作视频，配置关键点判定规则（如角度阈值、判定逻辑）。
    *   **价值**：实现动作标准的动态配置，无需修改代码即可新增支持的运动类型。

3.  **数据统计面板 (Statistics Panel)**
    *   **功能**：
        *   **用户画像分析**：基于饼状图展示用户的年龄分布、性别比例。
        *   **功能使用分析**：统计不同运动功能的使用频率分布。
        *   **多维度筛选**：支持今日、近七天、本月、今年、全周期的时间筛选。
    *   **价值**：辅助运营人员了解用户群体特征，制定针对性的健身内容。

4.  **运动记录管理 (Exercise Records)**
    *   **功能**：
        *   **记录查询**：查看用户的运动历史，包括运动时间、时长、类型。
        *   **效果分析**：统计正确个数、错误个数及正确率。
    *   **价值**：追踪用户运动效果，分析动作质量。

5.  **系统配置中心 (Configuration Center)**
    *   **功能**：
        *   **业务规则配置**：每日打卡上限、积分获取规则等。
        *   **系统开关**：维护模式开关、全局通知发布。
        *   **参数调优**：AI 识别阈值、超时时间等系统级参数配置。
    *   **价值**：提升系统的灵活性与可配置性，降低发版频率。

## 5. 模块关系说明

*   **运动分析链路**：
    `Client (Camera)` -> `UploadController` -> `FileStorageService` (暂存) -> `PoseController` -> `PoseAnalysisService` (计算) -> `AnalysisResult` (返回) -> `Client (Render)`
*   **AI 咨询链路**：
    `Client (Chat UI)` -> `AIChatController` -> `LLMService` (构建 Prompt) -> `DeepSeek API` -> `LLMService` (解析响应) -> `Client`
*   **管理运营链路**：
    `Admin Web` -> `AdminController` -> `SystemConfigService` / `UserService` -> `Database` -> `System Behavior Update`
*   **数据记录链路**：
    `Client` -> `WeChat Cloud SDK` -> `Cloud Function (SportHistory)` -> `SportRecordRepository` -> `Cloud Database` -> `Admin Dashboard` (读取分析)

## 6. 架构风险分析

| 风险点 | 风险描述 | 应对策略 |
| :--- | :--- | :--- |
| **实时性瓶颈** | 当前架构依赖图片上传至服务器分析，网络延迟与服务器单线程处理能力可能导致反馈滞后，影响用户体验。 | 1. 引入异步任务队列（Celery）处理非实时分析需求。<br>2. 探索将 MediaPipe 模型移植至小程序端（WebAssembly/TensorFlow.js），实现端侧零延迟推理。<br>3. 优化图片压缩算法，减少传输带宽占用。 |
| **并发性能风险** | Flask 开发服务器无法支撑高并发请求，且图像处理为 CPU 密集型任务，容易阻塞线程。 | 1. 生产环境使用 Gunicorn + Nginx 部署。<br>2. 服务端开启多进程/多线程模式。<br>3. 对高频请求实施限流熔断机制。 |
| **隐私数据泄露** | 用户上传的健身图像涉及个人隐私，若在传输或存储过程中泄露将引发法律风险。 | 1. 传输层强制启用 HTTPS 加密。<br>2. 服务器端实施“即用即焚”策略，分析完成后立即删除临时文件。<br>3. 敏感数据（如人脸）在分析前进行模糊化处理。 |
| **依赖服务不稳定性** | 强依赖 DeepSeek API 与微信云服务，若第三方服务宕机，相关功能将完全不可用。 | 1. 增加服务降级策略（如 AI 不可用时切换为本地规则库回复）。<br>2. 实施重试机制与超时控制。<br>3. 缓存常见问题的 AI 回复，减少 API 调用依赖。 |
